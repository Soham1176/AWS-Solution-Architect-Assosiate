# AWS Identity and Access Management (IAM)

## AWS IAM Overview (Important for AWS SAA)

**AWS Identity and Access Management (IAM)** is a core AWS security service that controls **who can access AWS resources and what actions they are allowed to perform**. It works based on two security concepts: **Authentication** (verifying identity) and **Authorization** (granting permissions). IAM acts like a **security gatekeeper for your AWS account**, ensuring that only authorized users, applications, or services can interact with AWS resources such as servers, storage, or databases. For the AWS SAA exam, it is important to understand the main IAM components: **Users, Groups, Policies, Roles, Access Keys, MFA, CLI, SDK, and security best practices**.

---

## IAM Users

An **IAM User** represents a **single physical person or application that needs access to AWS**. Each user has unique credentials such as a **console password** for logging into the AWS web interface and **access keys** for programmatic access. A common AWS guideline is **“One physical user = One IAM user”**, which allows better tracking of actions performed in the account. Instead of sharing credentials, every individual should have their own IAM user identity.

**Important exam points**

* Represents an **individual identity**
* Has **login credentials**
* Used for **human access to AWS**

---

## IAM Groups

An **IAM Group** is a collection of IAM users that share the same permissions. Instead of assigning permissions to every user individually, AWS recommends **assigning permissions to groups and adding users to those groups**. For example, a company might create groups like **Developers, Admins, and ReadOnlyUsers**. IAM groups simplify permission management and reduce configuration errors.

**Important exam points**

* Groups **contain users only**
* **Groups cannot contain other groups**
* Used for **permission management at scale**

---

## IAM Policies

**Policies** are **JSON documents that define permissions** in IAM. They specify which actions are **allowed or denied** on specific AWS resources. Policies can be attached to **users, groups, or roles**. A policy might allow a user to read objects from an S3 bucket but deny deleting them. Policies follow the principle of **least privilege**, meaning users should only receive the minimum permissions required to perform their tasks.

**Important exam points**

* Written in **JSON**
* Define **Allow or Deny actions**
* Attached to **Users, Groups, or Roles**

---

## IAM Roles

An **IAM Role** is an identity that provides **temporary permissions** to access AWS resources. Unlike users, roles **do not have permanent credentials**. Instead, they are **assumed by users, applications, or AWS services**. Roles are widely used when **AWS services need to interact with other AWS services securely**. For example, an EC2 instance may assume a role that allows it to access an S3 bucket without storing access keys on the server.

**Important exam points**

* Provides **temporary credentials**
* Used by **AWS services and applications**
* **More secure than access keys**

---

## IAM MFA (Multi-Factor Authentication)

**Multi-Factor Authentication (MFA)** adds an additional security layer to IAM authentication. Instead of relying only on a password, users must also provide a **temporary code generated by a device or authentication app**. Even if someone steals the password, they cannot access the account without the MFA code.

**Important exam points**

* Adds **extra security layer**
* Combines **password + authentication code**
* Recommended for **root account and IAM users**

---

## Access Keys

**Access Keys** are credentials used for **programmatic access to AWS services**. They consist of two parts: **Access Key ID and Secret Access Key**. These keys are used by tools like the AWS CLI or applications using SDKs to authenticate with AWS services.

**Important exam points**

* Used for **CLI and SDK access**
* Consist of **Access Key ID + Secret Access Key**
* Should be **kept secure and rotated regularly**

---

## AWS CLI

The **AWS Command Line Interface (CLI)** is a command-line tool that allows users to **manage AWS services using terminal commands instead of the web console**. Developers and administrators use CLI for automation, scripting, and faster resource management.

Example command:

```bash
aws s3 ls
```

This command lists S3 buckets in the account.

**Important exam points**

* Used for **automation**
* Requires **Access Keys for authentication**

---

## AWS SDK

An **AWS SDK (Software Development Kit)** allows developers to interact with AWS services using **programming languages such as Java, Python, JavaScript, or C++**. SDKs provide libraries that simplify communication between applications and AWS services.

Example use cases:

* Uploading files to S3
* Creating EC2 instances
* Sending messages to queues

**Important exam points**

* Used inside **applications**
* Supports multiple programming languages
* Uses **Access Keys or IAM Roles**

---

## Roles for AWS Services

AWS services often need to access other AWS resources securely. Instead of storing access keys, AWS recommends using **IAM roles for AWS services**. For example, an EC2 instance can assume a role that allows it to read data from an S3 bucket or write logs to CloudWatch.

**Important exam points**

* Used for **secure service-to-service communication**
* Eliminates need for **hardcoded credentials**
* Provides **temporary permissions**

---

## IAM Security Tools

IAM provides tools that help monitor and audit access in an AWS account.

Examples include:

**IAM Credential Report**

* Lists all IAM users
* Shows password usage and key status

**IAM Access Advisor**

* Shows which services a user actually used
* Helps remove unnecessary permissions

These tools help enforce the **least privilege principle**.

---

## IAM Best Practices

AWS recommends several best practices for securing IAM.

Key best practices include:

* **Do not use the root account for daily tasks**
* **Create IAM users for individuals**
* **Assign permissions through groups**
* **Enable MFA for better security**
* **Use roles for AWS services instead of access keys**
* **Rotate access keys regularly**
* **Never share IAM credentials**

Following these practices helps protect AWS environments from unauthorized access and configuration mistakes.
